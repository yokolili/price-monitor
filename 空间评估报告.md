# 价格观察站 — 空间资源评估报告

> 评估对象：`price-monitor` 网站（金价/石油/内存/硬盘/CPU 五类价格每日更新）
> 评估日期：2026-07-08
> 部署方式：GitHub Pages（gh-pages 分支，peaceiris 替换式部署）

---

## 一、实测：每次构建生成了什么、多大

完整跑一遍 `fetch.py + backfill.py(365) + build.py` 后实际产物：

| 产物 | 数量 | 单文件大小 | 小计 |
|---|---|---|---|
| `public/index.html`（首页） | 1 | 8 KB | 8 KB |
| `public/archive/YYYY-MM-DD.html`（每日归档） | 366 | ~6.9 KB | ~2.53 MB |
| `public/weekly/index.html`（周报） | 1 | 20 KB | 20 KB |
| `public/monthly/index.html`（月报） | 1 | 16 KB | 16 KB |
| `public/yearly/index.html`（年报） | 1 | 20 KB | 20 KB |
| **`public/` 合计（实际上线内容）** | — | — | **3.0 MB** |
| `data/history/YYYY-MM-DD.json`（原始数据） | 365 | 8 KB | 2.9 MB |

> ⚠️ **关键**：`data/history/` 与 `public/` 都已被 `.gitignore` 忽略，**不会进入 Git 仓库**。真正推送到 GitHub 的只有 `public/` 的内容（通过 gh-pages 分支）。

## 二、空间占用模型（最重要的一点）

### 当前架构的"自清洁"特性

`backfill.py` 每次都**重新生成截至今天往前 365 天的完整历史**（`range(1, 365)` 固定 364 天 + 当天 1 天），而不是在旧数据上追加。因此：

- 无论网站运行 1 天还是 10 年，`data/history` 始终是约 **365 个文件**，`public/archive` 始终是约 **366 个页面**。
- 每次构建产物 `public/` 大小**恒定 ≈ 3.0 MB**，不会随时间线性增长。

### 部署方式决定 GitHub 上"永久占用"多少

`peaceiris/actions-gh-pages` 默认 `force_orphan: true`，即 gh-pages 是**孤儿分支，每次完全覆盖**，不累积历史版本。

| 维度 | 当前真实情况 | 说明 |
|---|---|---|
| GitHub 永久存储占用 | **恒定 3.0 MB** | 每次部署是"整体替换"，不是每天 +3MB |
| 每日净新增存储 | **≈ 0** | 旧内容被新内容整体覆盖 |
| 每日上传流量（带宽） | ~3 MB/天 | 覆盖式上传，一年约 1.07 GB 出/入流量 |
| `main` 分支（源码） | < 100 KB | 不含 public/、data/ |

## 三、一年所需空间资源评估

| 场景 | 一年末存储空间 | 是否增长 |
|---|---|---|
| **当前实际（替换式部署）** | **3.0 MB（恒定）** | 不增长 |
| 假设改为"逐日累积"模式 | 3.0 + 365×6.9KB ≈ **5.5 MB** | 缓慢增长 |
| 假设累积运行 5 年 | ~15 MB | 仍极小 |

> 即使按最不利"逐日累积"估算，一年也仅新增约 **2.5 MB**，对 GitHub 是九牛一毛。

## 四、GitHub 提供的空间资源 vs 需求

| GitHub 限制项 | 额度 | 本站需求 | 满足度 |
|---|---|---|---|
| 单仓库建议大小（软限） | **1 GB** | 3.0 MB | ✅ 占用 0.3% |
| 单仓库硬限 | 5 GB | 3.0 MB | ✅ |
| 单个文件上限 | **100 MB** | 最大 20 KB | ✅ |
| GitHub Pages 单站点 | 建议 < 1 GB | 3.0 MB | ✅ |
| 免费账号仓库数 | 无限（个人） | 1 个 | ✅ |

**结论：GitHub 提供的空间资源完全满足，且绰绰有余。** 按当前恒定模型，网站运行 **300 年** 也才到 1 GB 软限；即便改成逐日累积，也要运行 **300+ 年** 才会触及限制。

## 五、补充：3 个备份空间（GitHub / GitLab / Gitee）

`backup.sh` 设计同步到 3 个远程。每个远程都会存一份 ~3.0 MB 的 `public/`：

| 备份位置 | 单份大小 | 3 份合计 |
|---|---|---|
| GitHub（gh-pages，已启用） | 3.0 MB | — |
| GitLab（待配置） | 3.0 MB | — |
| Gitee（待配置） | 3.0 MB | **≈ 9 MB** |

三个平台各 3 MB，合计不到 10 MB，均在各自免费额度内，毫无压力。

---

## 六、结论

1. **每日生成网页的存储成本 ≈ 0（净增）**，因为部署是整体替换式，GitHub 永久占用恒定在 **3.0 MB**。
2. **一年所需空间：恒定 3.0 MB**（替换式）或最多 **5.5 MB**（累积式），远小于 GitHub 的 1 GB 软限制。
3. **GitHub 空间资源完全满足**，无需任何扩容或付费升级。
4. 三个备份空间即使全部启用，总计也不到 10 MB，同样毫无压力。

> 唯一值得留意的是 **Actions 运行分钟数**（每天一次构建约 1–2 分钟，免费额度 2000 分钟/月），而非存储空间——空间方面可以完全放心。
